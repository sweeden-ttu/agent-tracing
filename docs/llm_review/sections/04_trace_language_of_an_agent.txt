The Trace Language of an Agent

This section is the formal core of the paper.  We define an agent
as a tuple of seven objects, three of which are derivative.  We
then define the agent's trace language and prove that the
definition satisfies the Chomsky hierarchy verbatim.  The
construction is austere and self-contained; nothing beyond
Section sec:preliminaries is assumed.

The CSV files are languages

We make explicit the empirical correspondence used throughout the
sequel.  The header row of
examples/rogii/traces/preprocessing/baseline_column_transformer/trace_language.csv
names twenty sub-agent swim lanes:

A_1 = data_downloader, , A_20 = dependency-graph-orchestrator,

followed by nine resource-envelope columns (chomsky_type, llm_model,
context_window_tokens, context_policy, long_term_memory,
ltm_growth_bound, slurm_partition, slurm_resources, gpu_device).

For each sub-agent A_i, the i-th swim-lane column of the file (skipping
empty cells) is a single string.  This string is one accepting run of
the orchestrator on the Rogii Wellbore Geology Prediction input.  The
trace language L(G) contains, in principle, infinitely many other
strings; the column records exactly the one that the recursive
enumeration emitted.

The same correspondence holds for each variant trace under
examples/rogii/traces/preprocessing/{variant}/trace_language.csv.
The baseline variant uses ColumnTransformer + LightGBM tokens; sibling
variants swap preprocessing tokens (typewell alignment, PS masks,
RobustScaler+log1p, parallel loaders, formation KNN) while preserving
the twenty-lane orchestrator skeleton.

Variant trace languages and critical review

We audited all eight roots matching /lustre/work/sweeden/agent-tracing*
for trace_language.csv artefacts that implement the Rogii Wellbore
Geology Prediction Kaggle competition
(rogii-wellbore-geology-prediction) under the scientific concept of
each variant's base paper (examples/rogii/BASE_PAPERS.md).  The
unified repository agent-tracing holds all six variant bundles under
examples/rogii/traces/preprocessing/{variant}/ together with
experiment_descriptor.json, ablation_plan.json, and paper PDFs.
Each branch worktree agent-tracing-trace-{baseline,typewell,ps,robust,
parallel,formation} carries a single variant CSV for isolated review;
agent-tracing-main mirrors baseline only.  Canonical traces for
critical review are the main-repo CSVs; baseline matches its worktree
byte-for-byte; the other five worktree copies are stale snapshots that
omit recent experiment-design tokens (configure_variant_ablation_flags,
design_ablation_factorial_grid, load_experiment_descriptor).

Every variant trace solves the same submission envelope: predict tvt
for each test id (columns id, tvt).  Training consumes per-well
horizontal-well CSVs with columns MD, X, Y, Z, ANCC, ASTNU, ASTNL,
EGFDU, EGFDL, BUDA, TVT, GR, TVT_input; typewell CSVs with TVT, GR;
test horizontal-well CSVs with MD, X, Y, Z, GR, TVT_input.  Shared
orchestrator lanes (data_downloader through dependency-graph-orchestrator)
bootstrap a Type-3-bounded DAG, enforce nested GroupKFold by well, train
LightGBM with objective regression and metric RMSE, and emit
slurm_episodic_trainer tokens only after Matador GPU binding.  Variant
theory enters in the feature_engineer, preprocessor, target_diagnostician,
and cv_orchestrator lanes---not in the trainer tail, which is
competition-shared.

Across variants, the trace-language theory claim holds at the
architectural level: each CSV is one accepting run of a
Goal-Driven Type-0 producer bounded by a Data-Driven Type-3 consumer,
and each variant's distinguishing operations appear in the
feature/preprocess lanes before the shared LightGBM tail.  The
remaining engineering debt is symbol-level fidelity between
ablation_plan.json factorial levels and the emitted alphabet Sigma_G,
and synchronising branch worktree CSVs with the canonical main-repo
traces before merge.  Full paper-to-token joins are machine-readable in
examples/rogii/traces/preprocessing/{variant}/experiment_descriptor.json
and human-readable in paper_refs.md.

The empirical signature of a Type-0 trace language

A trace language  is Type-0 iff  can
emit traces whose length grows without a recursive bound in the
input size, and whose contents reflect arbitrary recursive
enumerations of the operation alphabet.  In our empirical
setting, 's Type-0 status is witnessed by the
slurm_episodic_trainer column of
examples/rogii/traces/preprocessing/baseline_column_transformer/trace_language.csv (enumerated trace L_K), which contains a strictly
longer sequence (36 non-empty entries) than any column of
examples/rogii/traces/preprocessing/baseline_column_transformer/trace_language.csv (start language L_0) (which contains at most
fourteen).  The lengthening is the operational footprint of the
recursive enumeration: the start language emitted twelve seed
agents; the binding to the Type-3 verifier forced enumeration of
arbitrarily long check-train-validate cycles; the result was a
strictly more complex trace, materialised on disk by the
recursive Type-0 step.

The undecidability that Robinson's
theorem robinson1949definability,davis1961decision fixes
on the Type-0 class is therefore a real property of:
were it not for the Type-3 verifier, no algorithm would decide
whether a candidate trace lay in.  The verifier
is the entire reason the system is decidable in practice.
