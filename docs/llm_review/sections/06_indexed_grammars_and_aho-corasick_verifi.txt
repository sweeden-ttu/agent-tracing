Indexed Grammars and the Aho-Corasick Verifier

This section deepens the role played by Aho's two great
contributions to the theory of formal languages:
indexed grammars aho1968indexed, which fix the language
class strictly between context-free and context-sensitive, and
the Aho-Corasick aho1975corasick multi-pattern matcher,
which compiles a finite keyword set into a deterministic
linear-time recogniser.  The two papers were written eight years
apart, but in our framework they play complementary roles: the
indexed grammar is the natural home of the iterative repair loop
inside the Goal-Driven generator; the Aho-Corasick DFA is the
realisation of the Data-Driven verifier.

Why the iterative repair loop is indexed, not
context-free

In soporchestrator's trace we observe the cycle
verbatim
   invoke_planner
-> invoke_developer
-> invoke_reviewer
-> compare_score_to_advance_threshold(>=3)
-> [ if score < 3:
        retry_phase_on_low_score (loop)
        cap_iterations_at_max(=3) ]
-> invoke_summarizer_on_advance
-> emit_outcome(completefail).
verbatim
Three observations make this trace indexed rather than
context-free:
enumerate
The iteration counter is shared across three nested
sub-traces (planner / developer / reviewer); the same counter
appears synchronously in each.  This is a cross-serial
dependency, which a PDA cannot recognise.
The counter is bounded by a constant (3); this rules out
arbitrary linear-bounded behaviour.  The counter is not a tape, it
is an index.
Each invocation of the loop pushes a fresh index
configuration (the EDA bundle, the failed-test list, the
clamped score) that is consumed in inverse order during the
forwardartifacttonextphase step.
enumerate

These three properties are exactly the ones Aho identified in
1968 as motivating the indexed-grammar
extension aho1968indexed.  We therefore place
soporchestrator in the indexed zone.  The placement
sharpens, but does not contradict, the Type-1 placement of
taskorchestrator and kernelpipeline, whose
workspace genuinely grows with input size.

Why Aho-Corasick rather than a generic LLM judge

A generic LLM-as-judge could, in principle, also play the role of.  We deliberately chose Aho-Corasick for three
reasons:
enumerate
Class containment.  An LLM-as-judge is itself a
Type-0 agent.  Composing two Type-0 agents does not lower the
class.  An Aho-Corasick DFA is Type-3 and lowers the
verification budget to  by Theorem thm:closure.
Auditability.  The keyword set  is finite, human
readable, and version-controlled; the failure links and goto
table are derivable mechanically.  An LLM-as-judge offers no
such audit surface.
Soundness against the Type-0 generator.  If the
generator is permitted to enumerate arbitrary strings, the
verifier must be unable to be "talked into" accepting a
malformed trace.  A DFA cannot be talked into anything; it
either matches or it does not.  This is the precise sense in
which the Goal-Driven Type-0 agent is tamed by the
Type-3 verifier.
enumerate
